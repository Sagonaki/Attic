# Attic Phase 17 — Chase send_message p95 < 250 ms — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development.

**Goal:** Close the remaining gap to spec §1 (300 concurrent users, send_message **p95 < 250 ms**). Phase 16 reached 100 % OK / p95 = 909 ms; the tail is driven by GC pressure, per-send allocations, and backplane serialization — not DB or CPU headroom. Reduce it step-by-step with evidence-gated progression: measure after each change, stop as soon as p95 < 250 ms.

**Architecture:** Three targeted changes applied in order of expected ROI. Each task ends with an instrumented 300-user load run; we decline to add the *next* optimization if the previous one already met the target.

1. **MessagePack SignalR protocol.** Swap the default JSON hub protocol for MessagePack on both server (`AddMessagePackProtocol`) and load-test client. Shrinks every `MessageCreated` + `UnreadChanged` payload (fewer Redis bytes, less serialization CPU, less GC).
2. **Pool the fan-out work items.** `MessageFanoutWorkItem` is allocated per send → ~158 k/s during load. Move to `ObjectPool<T>` + rent the member-IDs list from `ArrayPool<Guid>` so the hot path stops churning young-gen allocations.
3. **Optional: dispatcher-thread / sharded fan-out.** If the first two don't close the gap, shard `MessageFanoutQueue` by `ChannelId` into N workers (N = CPU count or configured) so the Redis backplane is driven by multiple parallel pipes instead of one single-reader loop.

**Tech stack additions:** `Microsoft.AspNetCore.SignalR.Protocols.MessagePack` (already-in-Aspire-graph). `Microsoft.Extensions.ObjectPool` (built into ASP.NET Core). `System.Buffers.ArrayPool` (already in the BCL).

**Spec reference:** §1 (300-user target + 250 ms p95), §6.1-6.5 (SendMessage contract — must not change wire semantics visible to clients), §12.3/4 (testing philosophy).

## Phase 16 instrumentation — key drivers of the p95 tail

| Signal | Peak | Interpretation |
|---|---|---|
| GC allocation rate | **1.47 GB / 2 s** | MessageDto + work items + member-ID lists churn; gen-0 collections dominate |
| Working set | 6.8 GB | fan-out queue holds many items in flight under burst |
| ThreadPool queue length | 878 (brief) | bursty scheduling during fan-out spikes |
| Monitor lock contentions | 1289 / 2 s | Redis IConnectionMultiplexer writer pipe + SignalR group ops |
| Postgres active queries | 1 – 6 (steady) | DB is idle — not the bottleneck |
| Redis ops/s | ~158 k | headroom exists, but single-pipe serialization adds tail |

The two dominant drivers are **allocation churn** and **backplane serialization**. Tasks 1 and 2 attack both.

## Task 2: Pool MessageFanoutWorkItem via ObjectPool<T>

**Files:**
- Modify: `src/Attic.Api/Hubs/MessageFanoutQueue.cs` — convert to mutable class + pooled allocation
- Modify: `src/Attic.Api/Hubs/MessageFanoutService.cs` — return item to pool after drain
- Modify: `src/Attic.Api/Hubs/ChatHub.cs` — rent from pool before enqueue
- Modify: `src/Attic.Api/Program.cs` — register `ObjectPool<MessageFanoutWorkItem>`

Rationale: In Phase 16 we observed **1.47 GB/2 s peak allocation rate** mostly from MessageDto + work-item records. The DTO is harder to pool (crosses a scoped hub invocation boundary); the work item is entirely internal — rent → populate → enqueue → worker drains → return to pool.

- [ ] **Step 2.1: Convert `MessageFanoutWorkItem` from record → pooled class**

  Current (immutable record):

  ```csharp
  public sealed record MessageFanoutWorkItem(
      Guid ChannelId,
      MessageDto Message,
      IReadOnlyList<Guid> MemberIds);
  ```

  New (mutable class, reset on rent):

  ```csharp
  public sealed class MessageFanoutWorkItem
  {
      public Guid ChannelId;
      public MessageDto Message = default!;
      public Guid[] MemberIds = Array.Empty<Guid>();
      public int MemberCount;

      public void Reset()
      {
          ChannelId = default;
          Message = default!;
          // caller is responsible for returning MemberIds[] to the ArrayPool — see Task 3.
          MemberIds = Array.Empty<Guid>();
          MemberCount = 0;
      }
  }
  ```

  Note: `MemberIds` becomes `Guid[]` (not `IReadOnlyList<Guid>`) paired with an explicit `MemberCount` since `ArrayPool<Guid>` returns arrays that may be larger than the requested length.

- [ ] **Step 2.2: Write the `ObjectPool<MessageFanoutWorkItem>` policy**

  ```csharp
  file sealed class MessageFanoutWorkItemPolicy
      : Microsoft.Extensions.ObjectPool.PooledObjectPolicy<MessageFanoutWorkItem>
  {
      public override MessageFanoutWorkItem Create() => new();
      public override bool Return(MessageFanoutWorkItem obj)
      {
          obj.Reset();
          return true;
      }
  }
  ```

  Register in `Program.cs` after the `AddSingleton<MessageFanoutQueue>` line:

  ```csharp
  builder.Services.AddSingleton<Microsoft.Extensions.ObjectPool.ObjectPoolProvider,
                                Microsoft.Extensions.ObjectPool.DefaultObjectPoolProvider>();
  builder.Services.AddSingleton<Microsoft.Extensions.ObjectPool.ObjectPool<Attic.Api.Hubs.MessageFanoutWorkItem>>(sp =>
  {
      var provider = sp.GetRequiredService<Microsoft.Extensions.ObjectPool.ObjectPoolProvider>();
      return provider.Create(new MessageFanoutWorkItemPolicy());
  });
  ```

- [ ] **Step 2.3: Update `ChatHub.SendMessage` — rent from pool before enqueue**


- [ ] **Step 2.4: Update `MessageFanoutService` — return to pool after drain**

  ```csharp
  await foreach (var item in queue.Reader.ReadAllAsync(stoppingToken))
  {
      try
      {
          await hub.Clients.Group(GroupNames.Channel(item.ChannelId))
              .SendAsync("MessageCreated", item.Message, stoppingToken);

          var memberCount = item.MemberCount;
          var memberIds = item.MemberIds;
          var tasks = new Task[memberCount];
          for (int i = 0; i < memberCount; i++)
          {
              var memberId = memberIds[i];
              tasks[i] = Task.Run(async () =>
              {

                  await hub.Clients.Group(GroupNames.User(memberId))

              }, stoppingToken);

          await Task.WhenAll(tasks);

      {
          // Array comes from ArrayPool (Task 3). Order matters: return array THEN item.

      }
  ```

## Task 4 (conditional): Shard the fan-out queue by ChannelId

**Only run Task 4 if Checkpoint 2 did not hit p95 < 250 ms.**

**Files:**
- Modify: `src/Attic.Api/Hubs/MessageFanoutQueue.cs` — N inner queues, partition by `ChannelId.GetHashCode() % N`
- Modify: `src/Attic.Api/Hubs/MessageFanoutService.cs` — spawn N drain loops (one per shard)
- Modify: `src/Attic.Api/Program.cs` — configurable shard count (env var `FANOUT_SHARDS`, default = `Environment.ProcessorCount`)

Rationale: Phase 16's fan-out service is single-reader. Per-item parallelism via `Task.WhenAll` inside *one* item is fine but doesn't overlap *across* items. Sharding by channel keeps per-channel order (messages in the same channel go through the same shard) while letting different channels fan out concurrently.

- [ ] **Step 4.1: Introduce `FanoutOptions` with `ShardCount`**

  ```csharp
  public sealed class FanoutOptions
  {
      public int ShardCount { get; set; } = Environment.ProcessorCount;
  }
  ```

  Bind from configuration: `builder.Services.Configure<FanoutOptions>(builder.Configuration.GetSection("Fanout"));` and read env var `Fanout__ShardCount`.

- [ ] **Step 4.2: Rewrite `MessageFanoutQueue` as an array of channels**

  ```csharp
  public sealed class MessageFanoutQueue : IMessageFanoutQueue
  {
      private readonly Channel<MessageFanoutWorkItem>[] _shards;
      public ChannelReader<MessageFanoutWorkItem> ReaderFor(int shardIndex) => _shards[shardIndex].Reader;
      public int ShardCount => _shards.Length;

      public MessageFanoutQueue(IOptions<FanoutOptions> opts)
      {
          var n = Math.Max(1, opts.Value.ShardCount);
          _shards = new Channel<MessageFanoutWorkItem>[n];
          for (int i = 0; i < n; i++)
              _shards[i] = Channel.CreateUnbounded<MessageFanoutWorkItem>(new UnboundedChannelOptions
              {
                  SingleReader = true, SingleWriter = false, AllowSynchronousContinuations = false,
              });
      }

      public bool TryEnqueue(MessageFanoutWorkItem item)
      {
          var shard = (int)((uint)item.ChannelId.GetHashCode() % (uint)_shards.Length);
          return _shards[shard].Writer.TryWrite(item);
      }
  }
  ```
